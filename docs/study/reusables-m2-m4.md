# Reusables Harvest for M2-M4

> **Transport note (orchestrator, post-harvest):** this study was written against SPEC v1.1, before the M1 gate amendment. Wherever it says "sentinel (bulk)" for authoritative chunks/DONE, read **`eval_with_callback` pull via the bootstrap outbox** per SPEC §7.3 v1.2 (`plans/m1-bridge-findings.md`). The engine-side change is small: `emit()` routes bulk into `enqueueBulk` instead of sentinel segmentation. All selector/strategy/workflow citations are unaffected.

Target: M2 providers/adapters/engine, M3 workflow engine, M4 control pane UX. The original extension is the port source for provider DOM and workflow behavior; better-agent-terminal is MIT-reusable for Tauri host/layout patterns; tempo-term findings are `[clean-room]` patterns only.

## Co-work Pane Patterns

- Model each embedded agent web tool as a pane with independent profile/state, not as a special global singleton. Better-agent-terminal stores per-terminal agent preset, cwd, model/session metadata, history key, and activity in each terminal record (`refs/better-agent-terminal/renderer/src/stores/workspace-store.ts:341-400`, `refs/better-agent-terminal/renderer/src/stores/workspace-store.ts:843-900`).
- Preserve focus per workspace/pane so returning to a context restores the last active tool. Better-agent-terminal persists focused terminal on workspace switch and save (`refs/better-agent-terminal/renderer/src/stores/workspace-store.ts:79-150`, `refs/better-agent-terminal/renderer/src/stores/workspace-store.ts:874-885`).
- Keep inactive panes alive when their underlying process/session should continue. Better-agent-terminal renders all terminal panels and hides inactive wrappers (`refs/better-agent-terminal/renderer/src/components/WorkspaceView.tsx:1298-1318`); tempo-term documents the same keep-mounted pattern for inactive tabs `[clean-room]` (`refs/tempo-term/src/stores/tabsStore.ts:26-32`).
- Track session progress by a composite key that separates agent identity from workspace/path so multiple agents in the same context do not collide. Tempo-term applies this to agent transcript progress `[clean-room]` (`refs/tempo-term/src/modules/claude-progress/lib/progressStore.ts:6-20`, `refs/tempo-term/src/modules/claude-progress/lib/progressStore.ts:59-90`).
- Discover live session scope from pane state, not only initial workspace state. Tempo-term walks pane trees and prefers each pane's live cwd when collecting watched sessions `[clean-room]` (`refs/tempo-term/src/modules/claude-progress/lib/collectSessionCwds.ts:4-46`).
- Scope native/webview events to the owning window or pane label. Tempo-term preview creation emits title/navigation events to the owning window label `[clean-room]` (`refs/tempo-term/src-tauri/src/modules/preview.rs:82-115`).

No M4 scope commitment: these patterns are useful when adding post-M4 co-work panes such as Claude Code web, but M4 should only deliver provider panes and the central control UX.

## Porting Risk Notes

- Extension runtime lifetime assumptions do not transfer. The original checks `chrome.runtime.id`, calls cleanup on invalid context, and wraps `chrome.runtime.sendMessage` (`refs/multi-ai-chat/src/content/base.ts:43-60`). Desktop engine must treat page JS lifetime as bootId-scoped and bridge availability as bootstrap-scoped.
- Extension tab identity does not transfer. The original maps URL updates/removals to connection state using Chrome tab IDs (`refs/multi-ai-chat/src/background/service-worker.ts:37-62`). Desktop state must be keyed by provider webview label and bootId.
- Extension messaging broadcasts are different. The original relies on content-script messages reaching extension pages and avoids rebroadcasting to prevent duplicates (`refs/multi-ai-chat/src/background/service-worker.ts:120-133`). Desktop Rust bus must be dumb and at-most-once per bridge message.
- MutationObserver lifetime is per JS context and can miss SPA or hard reload transitions. Original observer is installed once on `document.body` (`refs/multi-ai-chat/src/content/base.ts:278-315`) with backup polling (`refs/multi-ai-chat/src/content/base.ts:317-344`). Desktop bootstrap must re-run on hard loads and re-push engine on stale/unknown status.
- Selector lookup must remain per-use. `queryFirst` searches selectors at call time (`refs/multi-ai-chat/src/content/base.ts:20-26`) and send/response code repeatedly queries DOM (`refs/multi-ai-chat/src/content/base.ts:98-112`, `refs/multi-ai-chat/src/content/base.ts:150-157`). Caching DOM nodes across SPA changes risks stale handles.
- DONE payload authority changes. Original `RESPONSE_DONE` rides extension messaging directly (`refs/multi-ai-chat/src/content/base.ts:264-275`) and `waitForResponse` resolves on any matching done (`refs/multi-ai-chat/src/background/service-worker.ts:152-162`). Desktop must resolve only on sentinel-assembled DONE per SPEC (`docs/SPEC.md:264-269`).
- Login/open behavior changes. Original `OPEN_LOGIN` creates a browser tab (`refs/multi-ai-chat/src/background/service-worker.ts:93-98`); desktop must navigate embedded provider webviews or system browser fallback for blocked Gemini (`docs/SPEC.md:336-345`).
- Service-worker CORS workaround changes. Original HackMD routes through MV3 service worker (`refs/multi-ai-chat/src/shared/hackmd.ts:21-31`, `refs/multi-ai-chat/src/background/service-worker.ts:412-439`). Desktop should use a Rust command, not preserve Chrome messaging.
